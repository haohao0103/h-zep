# Zep/Graphiti → HugeGraph 1.7.0 Adapter

Adapts [Zep](https://github.com/getzep/graphiti)'s **Graphiti** temporal-knowledge-graph
interaction flow onto a real **HugeGraph 1.7.0** backend via REST API.

> PoC redline compliant: real HugeGraph REST + real DeepSeek LLM, no simulation.

## 1. Goal

Graphiti (Zep's open-source engine) stores agent memory as a **bi-temporal
knowledge graph** — entities + edges carrying `valid_at` / `invalid_at`, with
automatic stale-fact invalidation. Its native backends are Neo4j / FalkorDB /
Neptune (all Cypher). This adapter proves the same interaction flow runs on
HugeGraph 1.7.0, which has **no Cypher engine**.

## 2. Adaptation approach

Graphiti's data-access layer (`nodes.py`, `edges.py`, `driver/operations/`) is
Cypher-string based: every save/get/delete is a Cypher template generated by
`node_db_queries.py` and executed via `driver.execute_query(cypher)`. HugeGraph
1.7.0 cannot run Cypher, so a drop-in `GraphDriver` subclass is not viable.

Instead we mirror Graphiti's **interaction flow** (the part that is pure Python)
and re-implement the persistence layer natively on HugeGraph REST:

| Graphiti concept | This adapter | Reused? |
|---|---|---|
| Data model (EntityNode / EpisodicNode / EntityEdge fields) | mirrored | ✓ schema |
| LLM extraction (entities + edges from an episode) | DeepSeek via OpenAI SDK | ✓ recipe |
| Bi-temporal invalidation (stale edge → `invalid_at`) | `_invalidate_conflicts` | ✓ logic |
| Hybrid retrieval (vector + fulltext + graph traversal, additive fusion) | `search()` | ✓ recipe |
| Cypher persistence (`MERGE`/`MATCH`/`CALL`) | HugeGraph REST + traverser | ✗ rewritten |

Files: `zep_hg/driver.py` (HugeGraph REST client), `zep_hg/embedder.py`
(local sentence-transformers), `zep_hg/engine.py` (the flow).

## 3. HugeGraph 1.7.0 API constraints (pitfalls hit & fixed)

These are non-obvious and cost the most debugging time:

1. **Gremlin script engine unavailable** — `/gremlin` returns
   `gremlin-groovy is not an available GremlinScriptEngine`. All reads/writes go
   through the REST API + `/traversers/*` endpoints instead.
2. **`CUSTOMIZE_STRING` forbids `primary_keys`** — use it without `primary_keys`;
   the vertex id IS the entity uuid (so "get by uuid" = "get by id", no
   secondary index needed).
3. **`edgelabel` with `frequency=MULTIPLE` requires `sort_keys`** — we use
   `sort_keys=["uuid"]` so multiple edges can share an `(outV, inV)` pair.
4. **All label properties are non-null by default** — must pass `nullable_keys`
   for optional fields (`invalid_at`, `expired_at`, `attributes`, …).
5. **Pre-existing property type clash** — the shared `hugegraph` graph already
   had `created_at` as `DOUBLE` (from an earlier PoC) and it cannot be dropped
   (referenced). We use `created_time` (TEXT) instead.
6. **Vertex update quirk** — `PUT /graph/vertices/"{id}"?action=append` (id
   wrapped in double quotes, `action=append` to merge properties).
7. **Edge create body fields** — `outV/inV/outVLabel/inVLabel`, **not**
   `source/target/source_label/target_label`.
8. **No usable vector index** — HugeGraph 1.7 lacks a vector index on par with
   Neo4j's `db.index.vector.queryNodes`. Embeddings are held in-process
   (`LocalEmbedder` cache) and vector search is a cosine scan; graph traversal
   and fulltext still hit HugeGraph.

## 4. End-to-end run

```
PYTHONPATH=. python run_zep_hugegraph.py   # HF_HUB_OFFLINE=1 if no network
```

Verified flow (DeepSeek `deepseek-chat`, HugeGraph `hugegraph` graph):

| Step | Result |
|---|---|
| `init_schema` (rebuild) | Entity / Episodic vertex labels, RELATES_TO / MENTIONS edge labels created |
| `add_episode` ×3 (2024-03 → 2025-06 → 2025-09) | 10 Entity vertices, 3 Episodic vertices, 6 RELATES_TO edges |
| Hybrid search "谁擅长图数据库运维" | #1 张明→图数据库 (0.30), #3 李华→HugeGraph (0.27) |
| Hybrid search "张明现在在哪家公司工作" | Globex edges rank above Acme |
| **Temporal query @ 2024-10-01** "张明在哪家公司" | #1 张明→Acme公司 (0.42) — correctly returns the *historical* employer |

The temporal query proves bi-temporal semantics: at 2024-10-01 only the Acme
edge is `valid_at ≤ point < invalid_at`, so Globex (valid 2025-06) is excluded.

## 5. Scope & next steps

- **Implemented**: add_episode (LLM extract → store), hybrid search, temporal
  point-in-time query, stale-edge invalidation logic (`_invalidate_conflicts`).
- **Not yet wired** (Graphiti has them, deferred for PoC): community detection,
  saga nodes, bulk episode ingestion, MCP server, cross-encoder reranker.
- **Vector search** is in-process (cosine scan). For production scale, add a
  FAISS sidecar or upgrade to a HugeGraph build with vector index support.
- **Graphiti-native path**: a `HugeGraphDriver(GraphDriver)` subclass that
  injects into Graphiti's `Graphiti(graph_driver=…)` requires a Cypher→REST
  translator covering ~30 query templates; deemed too fragile for PoC.
